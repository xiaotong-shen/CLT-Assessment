# CLT Assessment — Architecture & Cost Reference

> Last updated: 2026-05. Reflects the Phase 1 POC codebase.

---

## 1. System Overview

A multi-stage adaptive assessment platform for ESL/ELD level placement at a Canadian
private secondary school. Students sit a ~45–60 min computerised test covering
Reading, Listening, Grammar, and Writing. The MSAT (Multi-Stage Adaptive Test)
engine selects items in real time based on performance; on completion an engine
recommendation is produced and stored for specialist review.

```
Student browser
  │  intake form (bilingual EN / zh-Hans)
  │  adaptive test UI ─── item renderers (MC, Cloze, Essay, Listening)
  │  completion screen
  │
  ├─► Next.js API Routes (Vercel serverless, ca-central)
  │     /api/attempts          POST create attempt
  │     /api/attempts/[id]/next        GET next item (engine logic)
  │     /api/attempts/[id]/respond     POST record + auto-score response
  │     /api/attempts/[id]/override    POST specialist override
  │     /api/attempts/[id]/translate   POST zh-Hans translation
  │     /api/attempts/[id]/audit.json  GET full audit dump
  │     /api/items                     GET/PATCH item bank (staff)
  │
  ├─► PostgreSQL (Supabase, ca-central-1)
  │     via Drizzle ORM + postgres-js
  │
  ├─► Anthropic API (claude-haiku-4-5)
  │     Writing grader  ── on essay submit (mandatory)
  │     Batch translator ── on-demand by staff from report page
  │
Staff browser
  ├─► Queue  /queue         — all attempts, student name, course, flags
  ├─► Review /attempt/[id]/review — full response log + grading + override form
  └─► Report /attempt/[id]/report — family-facing PDF-ready report + zh-Hans toggle
```

## 4. Database Schema

Hosted on **Supabase PostgreSQL 16, `ca-central-1` region**. Migrations managed via
`drizzle-kit`. All IDs are CUID2 strings.

```
users                   Auth.js user store + role (student | specialist | admin)
accounts                OAuth accounts (Auth.js)
sessions                JWT session tokens (Auth.js)
verification_tokens     Magic-link tokens (Auth.js)

items                   Assessment item bank
  ├─ strand             reading | listening | grammar | writing
  ├─ format             mc-single | mc-multi | cloze | listening-mc | essay
  ├─ payload (jsonb)    format-specific content (passage, stem, options, etc.)
  └─ status             drafted | reviewed | live | retired

rubrics                 Writing assessment rubrics (traits with weights)
prompts                 Standalone writing prompts referencing a rubric

attempts                One per student assessment session
  ├─ intakeAnswers (jsonb)     name, DOB, L1, schoolingYears, l1LiteracySelfRating
  └─ recommendation (jsonb)   Recommendation object written on completion

attempt_items           Each item shown during an attempt
  ├─ strand, stage (route|target|confirm), level
  ├─ response (jsonb)   raw student answer; for essays includes grading sub-object
  └─ isCorrect          boolean; for essays: scoredLevel >= itemLevel

writing_responses       Standalone writing response table (future use)
recommendation_overrides  Specialist manual overrides with reason + original
audio_assets            TTS-generated audio clips for listening items (future)
```

**Key design choice:** Essay grading results are stored inline in
`attempt_items.response` as `{ text, grading: { scoredTraits, scoredLevel, ... } }`
rather than in the `writing_responses` table. This avoids a foreign-key dependency
on the `prompts` table and keeps the attempt audit trail self-contained.

## 5. MSAT Engine

The engine lives in `src/engine/` and has **zero imports from Next.js or the server
layer** — it is a pure TypeScript module that can run in any environment.

### Algorithm (Multi-Stage Adaptive Test)

Each strand (reading, listening, grammar, writing) goes through three stages:

```
Stage 1 — Route (4 items)
  Items at levels 2 and 4. Gets a coarse estimate of the student's range.

Stage 2 — Target (4 items)
  Items clustered around the estimated level from Stage 1.
  Narrows the estimate.

Stage 3 — Confirm (4 items)
  Items at the estimated level ± 1. Confirms placement.
  
→ estimatedLevel stored in strandProgress
```

`decide(state, nowMs)` returns either:
- `{ kind: "next-item", strand, stage, constraints: { levels } }` — pick an item
- `{ kind: "done", recommendation }` — all strands complete, emit recommendation

### Course Mapping

| Avg level | ESL course | ELD course |
|---|---|---|
| 1 | ESLAO | ELDAO |
| 2 | ESLBO | ELDBO |
| 3 | ESLCO | ELDCO |
| 4 | ESLDO | ELDDO |
| 5 | ESLEO | ELDEO |
| 6+ | Mainstream | — |

**Routing rule:** Course is determined by the *lowest* strand level, not the average.
A student with reading=5, writing=2 is placed at Level 2 (ESLBO) because writing
is the bottleneck.

### Stream Routing (ESL vs ELD)

ELD is indicated when:
- listening >> reading by ≥ 2 levels (oral language stronger than literacy)
- OR `l1LiteracySelfRating` ≤ 2 AND overall level ≤ 2

### Flags

| Code | Trigger | Severity |
|---|---|---|
| `uneven-profile` | Max strand level − min strand level > 1 | warn |
| `stage-3-ambiguous` | Confirm-stage accuracy 50–60% | info |
| `stream-eld` | ELD routing activated | info |
| `rushed` | Assessment completed in < 22 minutes | warn |
| `rapid-clicks` | ≥ 3 MC responses in < 2 seconds each | warn |
| `writing-blank` | Essay submitted with < 10 words | warn |
| `audio-skipped` | Listening item answered in < 3 seconds | warn |

## 6. AI / LLM Pipeline

### 6.1 Writing Grader

**File:** `src/server/writing-grader.ts`  
**Trigger:** Automatically when a student submits an essay response  
**Model:** `claude-haiku-4-5`  
**Prompt caching:** System prompt (rubric + scoring guide, ~450 tokens) cached with
`cache_control: { type: "ephemeral" }` — saves ~90% on input tokens for all essays
after the first in a worker lifecycle.

**Flow:**
```
1. Fetch rubric from DB (rubric-v1: taskFulfillment, linguisticRange,
   linguisticAccuracy, coherence — each weighted 1)
2. Build system prompt with rubric traits and Ontario STEP descriptors
3. Send essay text + prompt text to claude-haiku-4-5
4. Parse JSON response: { traits[{trait, score, rationale}], overallLevel, rationale }
5. Store in attempt_items.response as { text, grading: GradingResult }
6. Set isCorrect = (scoredLevel >= itemLevel) for engine advancement
```

**Error handling:** On API failure, stores `{ text, gradingError }` and sets
`isCorrect = true` (so the test can progress rather than stalling).

### 6.2 Batch Translator

**File:** `src/server/translate.ts`  
**Trigger:** On-demand — staff clicks "中文版" on the report page  
**Model:** `claude-haiku-4-5`  
**Caching:** Two-layer:
1. **In-process Map** keyed by `"zh-Hans:<sha256-16>"`  — zero API calls for repeated
   strings within the same worker lifetime
2. **Prompt caching** on system prompt (translator persona + format instructions)

**What gets translated (in one batch call):**
- Engine reasoning steps (up to 5 lines)
- Flag detail descriptions
- Essay trait rationales (one per scoring dimension)
- Model's overall writing rationale

**Format:** Input is a numbered list; output is a JSON array — robust to accidental
markdown fences.

### 6.3 Future: TTS Audio Generation (T-060)

**Not yet implemented.** See section 9 for full pipeline design.

**Planned model:** OpenAI TTS (`tts-1` or `tts-1-hd`)  
**One-time generation** — audio clips are generated offline and stored in Supabase
Storage. Students stream from a signed URL; no LLM calls during assessment.

## 8. Authentication Model

```
Students (role: "student")
  └─ Resend magic-link → email with one-time token
     → verified via NextAuth session
     → no password stored

Staff / Specialists (role: "specialist" | "admin")
  └─ Email + password (bcryptjs hash stored in users.image column for MVP)
     → JWT session with role embedded in token

Role-based access:
  student     → intake + attempt + complete pages only
  specialist  → + queue + review + report + item bank (read + patch)
  admin       → full access (same as specialist in Phase 1)
```

**Note:** Password hash is stored in `users.image` as a POC convenience — no
additional schema column needed. A Phase 2 cleanup would add a `password_hash`
column.

## 9. Data Flow: Student Assessment

```
1. Student navigates to /{locale}/intake
   → Fills bilingual intake form (name, DOB, L1, schooling years, L1 literacy)
   → POST /api/attempts { intake }
   → Server creates attempt row, snapshots live item IDs (for audit)
   → Returns { attemptId }
   → Browser redirects to /{locale}/attempt/{attemptId}

2. For each item:
   a. GET /api/attempts/{id}/next
      → buildAttemptState() replays DB rows to reconstruct engine state
      → decide(state) → { strand, stage, constraints }
      → DB query for items matching constraints, excluding already-shown
      → Picks least-exposed item (lowest nAttempts)
      → Records AttemptItem row (presentedAt, strand, stage, level)
      → Strips answer key via ClientItemSchema
      → Returns { item, stage, strand }
   
   b. Student answers; timer starts
      → Essay: auto-saved to localStorage every 1.5s
   
   c. POST /api/attempts/{id}/respond { itemId, response, timeMs }
      → scoreResponse() grades MC/cloze automatically
      → For essays: gradeEssay() calls Anthropic API (claude-haiku-4-5)
         → Returns { scoredTraits, scoredLevel, modelRationale }
         → isCorrect = scoredLevel >= itemLevel
      → Updates attemptItems row (response, isCorrect, timeMs)
      → Increments items.nAttempts (exposure counter)

3. On decide() returning { kind: "done" }:
   → Writes recommendation to attempts.recommendation (jsonb)
   → Sets attempts.status = "complete", finishedAt = now
   → Browser redirects to /{locale}/attempt/{id}/complete

4. Specialist reviews:
   → /queue — sees all attempts with student name, course code, flag count
   → /attempt/{id}/review — full response log, grading panel, override form
   → /attempt/{id}/report — clean printable report with optional zh-Hans translation
```

## 10. Key Design Decisions

| Decision | Rationale |
|---|---|
| **Engine is a pure module** | Can be unit-tested without DB/Next.js. Runs server-side only — students never see the adaptive logic. |
| **JWT strategy for sessions** | Credentials provider (staff) only works with JWT, not DB sessions. Magic-link tokens still stored in DB via adapter. |
| **Essay grading inline, not in `writing_responses`** | Avoids FK dependency on `prompts` table. Keeps audit trail self-contained in `attempt_items`. |
| **Grading on-submit, synchronous** | Students see a "Grading…" notice. Adds ~2–4s latency but keeps the flow simple — no background jobs needed in Phase 1. |
| **Translation on-demand** | Only specialists who need zh-Hans family reports pay the translation cost. Repeat translations hit the in-process cache for free. |
| **Dark mode removed from CSS** | Assessment tool must render consistently regardless of OS preference. Removed `@media (prefers-color-scheme: dark)` block entirely. |
| **`isCorrect` for essays = scoredLevel ≥ itemLevel** | Gives the engine a binary signal to advance/retreat levels. Not shown to students. |
| **Password stored in `users.image`** | POC convenience — avoids a schema migration for a feature (staff password login) used by < 10 people. Phase 2 adds `password_hash` column. |
| **Supabase ca-central-1** | Canadian data residency for student PII. Required for FIPPA/PHIPA compliance. |
| **cuid2 IDs** | Collision-resistant, URL-safe, no sequential enumeration risk. |

## 12. Upcoming: Audio Pipeline (T-060–T-062)

**Not yet implemented.** Requires specialist supervision of content before generation.

### Planned Flow

```
1. Author listening items in listening.json with transcript text
2. Run offline generation script: scripts/generate-audio.ts
   → Reads listening.json items
   → Calls OpenAI TTS API for each transcript
   → Uploads MP3 to Supabase Storage: audio/{itemId}.mp3
   → Gets a public or signed URL
   → Writes audioAsset record to DB (itemId, voice, url, durationMs)
3. Listening items now have a valid audioAssetId
4. GET /api/audio/[assetId] → 302 redirect to Supabase Storage URL
5. ListeningMc.tsx renders <audio src="/api/audio/{id}"> 
```

### TTS Provider Recommendation

**OpenAI TTS `tts-1-hd`** at $30/1M characters:
- 30 passages × ~200 chars = 6,000 chars → **$0.18 one-time**
- Voice: `nova` (clear, neutral) or `alloy` (neutral, slightly warmer)
- No ongoing cost — audio is pre-generated and cached in Supabase Storage

### Content Requirements (before generation)

- 30 listening passages authored by ESL specialist
- Passages graded Levels 1–5 (6 per level)
- Each passage: 50–200 words, read at natural pace
- Reviewed for cultural neutrality and Ontario curriculum alignment
- Audio clips should be 30–120 seconds each
