# Phase 1 Implementation Plan

This document is written to be handed off to a coding agent (Claude Sonnet,
Codex, etc.) one ticket at a time. Each ticket is self-contained with goal,
files to touch, acceptance criteria, and pitfalls. **Do not start a ticket
until its `Depends on` ids are merged.**

For the *why* behind any decision, refer to [`DESIGN.md`](DESIGN.md). This
document is the *what* and the *how*.

## 2. Conventions

A coding agent picking up a ticket should follow these by default.

- **TypeScript strict everywhere.** No `any`. If you reach for one, write a
  Zod schema instead.
- **Engine module is pure.** Files under `src/engine/` import nothing from
  `src/server/`, no DB, no fetch, no Date.now() (pass timestamps in). This
  is what makes the engine testable and Phase-3-swappable.
- **Server-only code lives under `src/server/`.** It uses `import "server-only"`
  at the top to fail loudly if it ever leaks to the client.
- **Validate at boundaries.** Every API route parses its body with a Zod
  schema before doing anything. Internal modules trust their types.
- **i18n keys are flat namespaces.** `intake.q.schoolingYears.label`, not
  nested objects. Easier to grep.
- **Tests live next to code** for unit tests (`__tests__/` subfolders) and
  in `tests/e2e/` for Playwright. Every ticket lists its tests in the
  acceptance criteria — coding agents should write them, not skip them.
- **Commit style**: simple lowercase one-liners. No Co-Authored-By trailer.
  No capitalization.
- **Env vars** are loaded through `src/lib/env.ts` (Zod-validated). Never
  use `process.env.X` directly in app code.
- **Errors at API boundary** return `{ error: { code, message } }` JSON with
  the right HTTP status. Never leak stack traces to the client.

## 3. The placement engine — concrete spec

This is the heart of Phase 1. A coding agent implementing the engine should
treat this section as the source of truth. Aligns with DESIGN.md §5–§6.

### 3.1 Types

```ts
// src/engine/types.ts

export type Strand = "reading" | "listening" | "grammar" | "writing";
export type Stream = "ESL" | "ELD";
export type Stage = "route" | "target" | "confirm";

/** 1..5 = ESL/ELD level; 6 = mainstream-ready (STEP step 6) */
export type Level = 1 | 2 | 3 | 4 | 5 | 6;

export interface ItemRef {
  id: string;
  strand: Strand;
  level: Level;            // authored difficulty
  subskill: string;
  format: ItemFormat;
}

export type ItemFormat =
  | "mc-single"
  | "mc-multi"
  | "cloze"
  | "matching"
  | "short-answer"
  | "essay"
  | "listening-mc";

export interface Response {
  itemId: string;
  level: Level;            // copied from item at presentation time
  correct: boolean;        // for essay: rubric-derived boolean (>= target)
  timeMs: number;
  stage: Stage;
}

export interface IntakeAnswers {
  l1: string;              // ISO 639-1 ideally
  yearsOfSchooling: number;
  l1LiteracySelfRating: 1 | 2 | 3 | 4 | 5; // 1 = "I can't read in any language"
  ageYears: number;
}

export interface AttemptState {
  attemptId: string;
  intake: IntakeAnswers;
  responses: Response[];
  /** per-strand stage progress */
  strandStages: Record<Strand, Stage | "done">;
}

export interface NextItemDecision {
  kind: "next-item";
  strand: Strand;
  stage: Stage;
  /** filter the item bank with these constraints */
  pickFrom: { levels: Level[]; subskills?: string[] };
}

export interface DoneDecision {
  kind: "done";
  recommendation: Recommendation;
}

export type Decision = NextItemDecision | DoneDecision;

export interface Recommendation {
  stream: Stream;
  course: string;          // e.g. "ESLCO"
  perStrandLevel: Record<Strand, Level>;
  stepProfile: Record<Strand, Level>; // identical to perStrandLevel for MVP
  flags: Flag[];
  reasoning: string[];     // human-readable trace, EN
}

export interface Flag {
  code: string;            // see flags.ts for list
  severity: "info" | "warn" | "review";
  detail: string;
}
```

### 3.2 MSAT algorithm (reading/listening/grammar)

Implement in `src/engine/msat.ts`:

1. **Stage `route`**: present 4 items at level 3 spanning subskills.
   Compute accuracy. If ≥ 75% → upper track (target levels {4,5}); if
   ≤ 25% → lower track ({1,2}); else mid track ({3}).
2. **Stage `target`**: present 4 items inside the track range.
   Compute estimated level using the §5.3 rule:
   ```
   estimated_level = max d such that
     accuracy_at_d_or_below >= 0.7 AND accuracy_at_d+1 < 0.5
   ```
   If denominator is empty for a check, treat that check as failed
   (conservative).
3. **Stage `confirm`**: present 2 items at `estimated_level - 1` and 2 at
   `estimated_level + 1`. If items at `+1` show ≥ 50% accuracy *and* items
   at `-1` show 100% accuracy → bump the estimate up by 1. If items at
   `-1` show < 100% → bump down by 1. Otherwise keep.

### 3.3 Writing strand scoring

Single prompt at the level the engine has currently estimated for the
student (use the average of reading + listening + grammar at the time the
writing prompt is presented, rounded down). The grader (§5) returns four
trait scores 1–5; final writing level = `round(mean(traits))`.

### 3.4 Stream routing (ESL vs ELD)

Implement in `src/engine/stream-router.ts`. Mirrors DESIGN.md §6.1:

```ts
function selectStream(state: AttemptState): { stream: Stream; flags: Flag[] }
```

ELD if any of:
- `intake.yearsOfSchooling < expectedForAge(intake.ageYears) - 2`
  AND `intake.l1LiteracySelfRating <= 2`
- `levelOf("listening") - levelOf("reading") >= 2`

Otherwise ESL. **Always** flag ELD-routed attempts with
`{ code: "stream-eld", severity: "review", ... }` for specialist review.

### 3.5 Course mapping

Implement in `src/engine/course-mapper.ts`. Lowest of the four strands
maps to course code per the DESIGN.md §6.2 table. Level 6 → "Mainstream
English (no ESL placement needed)".

### 3.6 Flags

Implement in `src/engine/flags.ts`. Triggers from DESIGN.md §11.1:

| Code               | When                                                |
|--------------------|------------------------------------------------------|
| `uneven-profile`   | `max(strand level) - min(strand level) > 1`          |
| `stage-3-ambiguous`| Any strand's confirm stage didn't move the estimate, but split was 50/50 |
| `stream-eld`       | Stream router selected ELD                            |
| `rushed`           | Total time < 50% of expected                          |
| `rapid-clicks`     | ≥ 3 items answered in < 2000ms                        |
| `writing-blank`    | Writing < 20 words at estimated level ≥ 3             |
| `audio-skipped`    | Listening item answered without playback event        |

Each flag with `severity: "review"` puts the attempt in the specialist
queue.

### T-041: Attempt lifecycle API ✅

**Goal:** Three endpoints that drive the whole test:
1. `POST /api/attempts/:id/next` — engine returns a `NextItemDecision` or
   `DoneDecision`. If next, server picks an item from the bank matching
   the constraints, records a `attempt_items` row with `presented_at`,
   returns the (sanitized — no answer key!) item payload.
2. `POST /api/attempts/:id/respond` — body: `{ itemId, response, timeMs }`.
   Server scores deterministically (MC), stores result on the
   `attempt_items` row.
3. `POST /api/attempts/:id/complete` — server runs `summarize(state)`,
   stores `recommendation_jsonb`, marks attempt `status: "complete"`.

**Effort:** L
**Depends on:** T-020, T-040

**Files:**
- `src/app/api/attempts/[id]/next/route.ts`
- `src/app/api/attempts/[id]/respond/route.ts`
- `src/app/api/attempts/[id]/complete/route.ts`
- `src/server/attempts.ts` (the read/write side)
- `src/server/items.ts` (the item picker)

**Approach:**
1. Item picker accepts `{ levels, subskills?, excludeItemIds }` and
   returns one item randomly weighted by inverse exposure (`n_attempts`).
2. Engine state is rebuilt on each request from `attempt_items` —
   stateless API, source of truth in DB.
3. `respond` is idempotent on `(attemptId, itemId)`.

**Acceptance:**
- [ ] E2E test: a synthetic "Level 3 student" runs through and gets
      ESLCO at the end
- [ ] `respond` called twice for same item doesn't double-score
- [ ] `next` after `complete` returns 409
- [ ] Attempt resumes correctly if browser closes mid-test

**Status:** Done. `src/server/attempts.ts` implements full lifecycle:
`createAttempt`, `buildAttemptState` (stateless replay), `getNextItem`
(ClientItemSchema strips answer keys), `recordResponse` (idempotent),
`getAuditJson`. API routes: `POST /api/attempts`, `GET …/next`,
`POST …/respond`, `GET …/audit.json`. Auto-completes attempt when engine
returns `done`. Audit endpoint gated to specialist/admin roles.

**Pitfalls:** Never include the answer key in the `next` response payload.
The MC payload sent to the client has `options` only — `correctOptionId`
is server-side.

## Epic E9 — Writing strand

### T-080: Anthropic SDK + caching setup ✅

**Goal:** Reusable `gradeWritingResponse({ text, level, rubricVersion })`
that calls Claude with prompt-cached rubric + anchor papers.

**Status:** Done — `src/server/writing-grader.ts` implemented with Claude Haiku via `@anthropic-ai/sdk`, rubric fetched from DB, prompt caching on system prompt, returns per-trait `TraitScore[]` + `scoredLevel` + `modelRationale`.

**Effort:** M
**Depends on:** T-002

**Files:**
- `src/grading/writing.ts`
- `src/grading/rubric.ts` (rubric data, anchor papers per band)
- `src/grading/__tests__/writing.test.ts` (with mocked SDK)

**Approach:**
1. Rubric is a constant in `rubric.ts` — four traits, descriptors per band.
2. Anchor papers: 3 per level (1, 2, 3, 4, 5) with rationale. Specialist
   provides; engineering wires them in.
3. Prompt structure:
   - System: rubric + anchor papers (cached)
   - User: candidate response
   - Output: structured JSON (use `tools` with a single
     `submit_score` tool to force JSON shape).
4. Pin model: `claude-sonnet-4-6` (or current at build time). Store the
   model id on `writing_responses.model`.

**Acceptance:**
- [ ] Mocked SDK returns valid JSON; `gradeWritingResponse` parses and
      validates with Zod
- [ ] Cache hit on second call (verify by inspecting usage block)
- [ ] Test asserts an obvious Level-1 response gets `traits.range <= 2`

**Pitfalls:** Cache requires the system prompt to exceed the cache
threshold (~1024 tokens). Anchor papers help reach this naturally.
